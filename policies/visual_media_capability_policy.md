# Policy: visual_media_capability_policy

## Goal

Define how Atlas should evaluate image, video, branding and visual-review capabilities without turning the factory into an asset-generation runtime.

This policy adapts the useful pattern from `claude-vibecoding`: visual direction first, evidence before PASS, and fallback paths that do not block the project when media generation is unavailable.

## Current stance

- Default: advisory-only.
- Real image generation: blocked until explicitly approved for a derived project.
- Real video generation: blocked until explicitly approved for a derived project.
- Visual analysis: allowed only as read-only review over local evidence supplied by the user or generated by an approved workflow.
- MCP/browser automation: blocked until a separate readiness decision approves it.
- Atlas core must not store generated brand assets for derived projects.

## Capability decisions

| Capability | Current Atlas coverage | Recommended posture | Reason |
| --- | --- | --- | --- |
| Branding review | Covered by design intelligence skills and policies | Keep active | Low risk and evidence-based. |
| Landing review | Covered by `design_intelligence_audit` and quality gates | Keep active | Already improves public-readiness decisions. |
| Image analysis | Partially covered through manual evidence review | Advisory-only | Useful when the user provides screenshots, but no external runtime is required by default. |
| Image generation | Not covered as runtime | Watchlist | Reference repo uses API keys and fallbacks; this belongs in derived projects, not Atlas core. |
| Video review | Not covered as runtime | Watchlist | Browser/video evidence can be expensive and brittle; keep manual unless a real need appears. |
| Video generation | Not covered as runtime | Defer | Requires external services, keys, long-running jobs and fallbacks. |
| Pixel/browser QA | Static audits only today | Watchlist | Valuable later, but real browser automation increases hang and MCP risk. |

## Rules

Atlas may recommend visual or media tooling only when all are true:

- the local design audit cannot answer the question with existing evidence
- the project has explicit visual direction or a user-provided reference
- the requested capability is tied to a concrete outcome
- the tool can return evidence, warnings and next action
- the user approves any external call, generated asset or config change

Atlas must not recommend visual or media tooling when:

- the request is only aesthetic preference without a concrete project goal
- the local repo and policies already provide enough evidence
- a tool would require credentials for a low-value task
- the output would be stored in Atlas core instead of a derived project
- the tool could hang without timeout or structured failure output
- the capability duplicates existing design-intelligence checks

## Evidence requirements

Any visual or media PASS must cite evidence:

- source file, screenshot path, or user-provided reference
- check name that produced the result
- status: `pass`, `warning`, `fail` or `skipped`
- reason for `skipped`
- explicit next action

If evidence is missing, the correct state is `skipped` or `needs_improvement`, not PASS.

## Image generation rule

Atlas can propose image generation later as a derived-project capability, but only after a separate approval step defines:

- intended asset
- target project path
- provider
- fallback
- cost or free-tier limits
- timeout
- where the generated asset will live
- validation plan

No image provider is approved by default.

## Video generation rule

Video generation must never block readiness. If a future project requests video, Atlas should prefer:

1. CSS or static-image fallback
2. short generated motion only with explicit approval
3. no video if the value is mostly decorative

The reference repo's useful pattern is the fallback discipline, not the specific video API.

## Integration guidance

- `quality_gate_report`: keep reporting public readiness from local evidence first.
- `priority_engine`: do not prioritize media generation unless a failed/warning check explicitly requires visual evidence.
- `prompt_builder`: when visual research is needed, ask for visual direction, audience, mood and originality before suggesting external tools.
- `external_tool_policy`: remains the parent rule for any external source or MCP.

## Future watchlist

Potential future additions, still not approved:

- `visual-evidence-brief` skill for collecting screenshots and references manually
- `media-readiness-report` helper that checks whether an image/video request has enough inputs
- read-only browser screenshot adapter after explicit readiness review
- external image or video provider profiles with default-deny policy
